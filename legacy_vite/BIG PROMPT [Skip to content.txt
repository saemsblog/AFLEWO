What is a payment gateway?
A payment gateway is a secure bridge between an online store and a bank. 

Think of it as a digital cashier that facilitates secure online purchases, or as an equivalent of a point-of-sale (POS) terminal.

It facilitates the authorization and processing of credit card transactions and other forms of electronic payments for both online and offline businesses. 

The gateway collects payment details from customers, verifies their authenticity, and communicates with the financial institutions involved to confirm whether the transaction can proceed

How Does a Payment Gateway Work?
The operation of a payment gateway involves several key steps:

Customer Information Collection: When a customer initiates a purchase, they provide their payment details, such as credit card information, on the merchant’s website or mobile app.
Data Encryption: To ensure security, the payment gateway encrypts the sensitive payment information, safeguarding it from unauthorized access during transmission.
Request for Authorization: The encrypted payment data is transmitted to the payment processor or acquiring bank, which forwards it to the customer’s card-issuing bank for verification.
Approval or Decline of Transaction: The issuing bank assesses whether the customer has sufficient funds and verifies the legitimacy of the transaction. It then sends back an approval or denial response through the same secure channels to the merchant.
Transaction Finalization: If approved, the transaction is finalized by transferring funds from the customer’s account to the merchant’s account, completing the payment process.
Importance in e-Commerce
Thus payment gateways play a crucial role in e-commerce by improving security and simplifying the transaction process. 

They accommodate a variety of payment methods, such as credit cards, debit cards, digital wallets, and even cryptocurrencies, providing convenience and flexibility for both merchants and customers.  

Also, by securely handling sensitive customer data, payment gateways foster trust and confidence, which can help lower cart abandonment rates during online shopping.

How to Choose the Best Payment Gateway in Kenya
When selecting a payment gateway, consider these key factors:

Credibility: Look for providers with a solid track record.
Safety: Prioritize gateways with robust security features.
Payment Method Compatibility: Ensure support for various payment options.
Integration Capabilities: Ensure that the gateway can easily integrate with existing e-commerce platforms or POS systems.
Transaction Fees: Understand the fee structure associated with each transaction, as this can impact profitability.

## 3) Payments architecture: M-Pesa (Daraja), Paybill, M-Changa, fallback (Stripe/Flutterwave)

### Payment goals

1. Accept Kenyan donors via M-Pesa (Paybill / STK Push / C2B).
2. Accept cards and international donors (Stripe + optional Flutterwave for local methods).
3. Offer M-Changa (crowdfunding) links/embed for campaigns.
4. Provide flawless end-to-end verification and reconciliation; no “fake success” acceptance.

### Recommended approach (production)

* **Primary for Kenyan donors:** Direct **M-Pesa C2B / Paybill** integration via Safaricom Daraja (server-side). Provide fallback via **Flutterwave** (if you prefer aggregator). 
* **International / cards:** **Stripe** for cards; optionally set up Stripe→M-Pesa via Stripe Africa if needed. 
* **Crowdfunding (M-Changa):** Embed M-Changa campaign pages or link out; also capture a webhook-confirmation from M-Changa to record donations in Supabase (M-Changa may provide callbacks or CSV exports to import).
* **All payment callbacks → your webhook endpoint** (hosted on Node server / serverless) that performs verification + atomic DB writes + idempotency checks + notification.

### Payment flow patterns (simplified)

**A. M-Pesa C2B (Paybill validation + confirmation)**

1. Donor sends M-Pesa to AFLEWO Paybill number with AccountReference or STK push.
2. Safaricom posts to your **validation** endpoint (expect to respond quickly with accepted/rejected).
3. Safaricom posts to your **confirmation** endpoint with transaction details once payment clears.
4. Your backend: validate payload signature (if available), check amount, check AccountReference (match event/donor), **create transaction record with transaction_id from Safaricom**, mark it received, trigger email/SMS receipts, and queue reconciliation.

**B. STK Push (Lipa na M-Pesa Online)**

1. Your server requests STK Push from Daraja (requires OAuth token).
2. Daraja triggers customer phone prompt for PIN.
3. Daraja sends asynchronous callback with the CheckoutRequestID and final result.
4. Same verification & idempotency flow as C2B. 

**C. Stripe (cards)**

1. Use Stripe Checkout or Elements to collect card.
2. Stripe provides webhooks (payment_intent.succeeded).
3. Verify webhook signature and write transaction atomically.

## 4) Airtight backend verification procedures (concrete)

Follow **three layers**: *Authentication*, *Idempotency & atomic DB writes*, *Reconciliation & retention*.

### A. Authentication & trust of callbacks

* **Daraja OAuth**: server obtains access token using consumer key/consumer secret. Store token securely in env and refresh when near expiry. Never expose these to client. 
* **Webhook verification**: if provider signs webhooks, verify signature header (HMAC) using provider shared secret. If no signature (some banks), enforce additional checks: source IP whitelist, TLS client certificate, or require provider to sign. Log raw payload for audit. 
* **TLS only:** Require HTTPS, strong ciphers, certificate monitoring.

### B. Idempotency & race conditions

* Every incoming callback must include a *provider transaction id* (e.g., MpesaReceiptNumber or Stripe id). Use that as the idempotency key. DB writes must be wrapped in a single transaction with INSERT ... ON CONFLICT DO NOTHING/UPDATE keyed on that id. Sample SQL schema below. 

sql
CREATE TABLE payments (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  provider text NOT NULL,
  provider_tx_id text NOT NULL UNIQUE, -- e.g., MpesaReceiptNumber or Stripe id
  status text NOT NULL,
  amount numeric(12,2) NOT NULL,
  currency text NOT NULL,
  metadata jsonb,
  received_at timestamptz,
  created_at timestamptz DEFAULT now()
);


Server logic (pseudocode):

ts
// Express example for webhooks
app.post('/webhooks/mpesa/confirmation', async (req, res) => {
  const payload = req.body;
  const txId = payload.MpesaReceiptNumber;
  await db.query(
    `INSERT INTO payments(provider, provider_tx_id, status, amount, currency, metadata, received_at)
     VALUES($1,$2,$3,$4,$5,$6,$7)
     ON CONFLICT (provider_tx_id) DO UPDATE
       SET status = EXCLUDED.status
     RETURNING *;`,
    ['mpesa', txId, payload.ResultDesc, payload.Amount, 'KES', payload, now()]
  );
  res.json({ ResultCode: 0, ResultDesc: 'Accepted' }); // quick response to provider
});


### C. Quick 200 to provider — async verification & processing

* **Respond fast** to provider (within 2–5 seconds with success code). Do heavy work (email, notification, ledger posting) asynchronously via job queue (BullMQ). This prevents timeouts and retries. 

### D. Full reconciliation

* **Daily reconciliation job**:

  * Pull bank statements (CSV) & M-Pesa Paybill statement (CSV/API).
  * Compare provider transaction ids and amounts with payments table.
  * Create exceptions report: missing confirmations, mismatched amounts, reversals.
  * Keep audit trail of reconciliation actions (who resolved, notes). 

### E. Webhook replay / retry safety

* Store raw webhook payloads and headers in raw_webhooks table with status pending/processed/failed. If processing fails, flag and retry with exponential backoff. Never drop raw payloads: they're evidence.

## 8) Webhook verification sample (Node.js / Express / TypeScript)

ts
import crypto from 'crypto';

function verifySignature(rawBody: Buffer, headerSignature: string, secret: string) {
  const hmac = crypto.createHmac('sha256', secret);
  hmac.update(rawBody);
  const expected = hmac.digest('hex');
  return crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(headerSignature));
}

app.post('/webhook/stripe', express.raw({type: 'application/json'}), (req, res) => {
  const sig = req.headers['stripe-signature'] as string;
  if(!verifySignature(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET)) {
    res.status(400).send('Invalid signature');
    return;
  }
  // parse and process...
});

      const raw = await request.text();

// Cloudflare Worker - mpesa webhook + STK push + token caching
// Build: tsc -> dist/index.mjs (or use esbuild). Keep minimal dependencies.

interface Env {
  KV_TOKENS: KVNamespace;
  KV_IDEMPOTENCY: KVNamespace;
  SUPABASE_URL: string;
  SUPABASE_SERVICE_ROLE: string;
  DARAJA_CONSUMER_KEY: string;
  DARAJA_CONSUMER_SECRET: string;
  DARAJA_SHORTCODE: string;
  DARAJA_PASSKEY: string;
  DARAJA_CALLBACK_BASE: string;
}

const DARAJA_OAUTH_URL = "https://api.safaricom.co.ke/oauth/v1/generate?grant_type=client_credentials";
const DARAJA_STK_URL = "https://api.safaricom.co.ke/mpesa/stkpush/v1/processrequest";

function nowIso() { return new Date().toISOString(); }

async function getDarajaToken(env: Env) {
  const kvKey = "daraja_token";
  const cached = await env.KV_TOKENS.get(kvKey, "json") as { token?: string; expires_at?: string } | null;
  if (cached && cached.token && cached.expires_at && new Date(cached.expires_at) > new Date()) {
    return cached.token;
  }
  const auth = btoa(`${env.DARAJA_CONSUMER_KEY}:${env.DARAJA_CONSUMER_SECRET}`);
  const res = await fetch(DARAJA_OAUTH_URL, {
    headers: { "Authorization": `Basic ${auth}` }
  });
  if (!res.ok) throw new Error("Daraja token fetch failed");
  const body = await res.json();
  // expiry: assume 3500s to be safe
  const expires_at = new Date(Date.now() + 3500 * 1000).toISOString();
  const tokenObj = { token: body.access_token, expires_at };
  await env.KV_TOKENS.put(kvKey, JSON.stringify(tokenObj));
  return body.access_token;
}

function base64Passkey(shortcode: string, passkey: string) {
  const timestamp = new Date().toISOString().replace(/[-:TZ.]/g, "").slice(0,14); // YYYYMMDDHHMMSS
  const raw = shortcode + passkey + timestamp;
  return { password: btoa(raw), timestamp };
}

async function writeRawWebhookToSupabase(env: Env, provider: string, payload: any, headers: Record<string, string>) {
  const res = await fetch(`${env.SUPABASE_URL}/rest/v1/raw_webhooks`, {
    method: "POST",
    headers: {
      "apikey": env.SUPABASE_SERVICE_ROLE,
      "Authorization": `Bearer ${env.SUPABASE_SERVICE_ROLE}`,
      "Content-Type": "application/json",
      "Prefer": "return=representation"
    },
    body: JSON.stringify({ provider, payload, headers, received_at: nowIso() })
  });
  // ignore res for now; but log on non-ok
  if (!res.ok) {
    const txt = await res.text();
    console.error("Failed to write raw webhook to supabase:", txt);
  }
}

// STK initiate handler
async function handleStkInitiate(req: Request, env: Env) {
  const body = await req.json();
  const { amount, phone, reference, accountReference } = body;
  if (!amount || !phone) return new Response("missing", { status: 400 });
  // create pending payment in Supabase via service role
  const insert = await fetch(`${env.SUPABASE_URL}/rest/v1/payments`, {
    method: "POST",
    headers: {
      "apikey": env.SUPABASE_SERVICE_ROLE,
      "Authorization": `Bearer ${env.SUPABASE_SERVICE_ROLE}`,
      "Content-Type": "application/json",
      "Prefer": "return=representation"
    },
    body: JSON.stringify([{
      provider: "mpesa", provider_tx_id: null, reference: reference || `AFLEWO-${Date.now()}`,
      amount, donor_phone: phone, status: "PENDING", metadata: { accountReference: accountReference || null }, created_at: nowIso()
    }])
  });
  if (!insert.ok) {
    const t = await insert.text();
    return new Response(t, { status: 500 });
  }
  const [payment] = await insert.json();
  const token = await getDarajaToken(env);
  const { password, timestamp } = base64Passkey(env.DARAJA_SHORTCODE, env.DARAJA_PASSKEY);
  const payload = {
    BusinessShortCode: env.DARAJA_SHORTCODE,
    Password: password,
    Timestamp: timestamp,
    TransactionType: "CustomerPayBillOnline",
    Amount: amount,
    PartyA: phone.replace(/^0/, "254"), // normalize
    PartyB: env.DARAJA_SHORTCODE,
    PhoneNumber: phone.replace(/^0/, "254"),
    CallBackURL: `${env.DARAJA_CALLBACK_BASE}/webhooks/mpesa/stk_callback`,
    AccountReference: accountReference || payment.reference,
    TransactionDesc: `AFLEWO donation ${payment.reference}`
  };
  const resp = await fetch(DARAJA_STK_URL, {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${token}`,
      "Content-Type": "application/json"
    },
    body: JSON.stringify(payload)
  });
  const bodyResp = await resp.json();
  // store CheckoutRequestID into payments metadata
  if (resp.ok && bodyResp.CheckoutRequestID) {
    await fetch(`${env.SUPABASE_URL}/rest/v1/payments?id=eq.${payment.id}`, {
      method: "PATCH",
      headers: {
        "apikey": env.SUPABASE_SERVICE_ROLE,
        "Authorization": `Bearer ${env.SUPABASE_SERVICE_ROLE}`,
        "Content-Type": "application/json"
      },
      body: JSON.stringify({ metadata: { ...payment.metadata, CheckoutRequestID: bodyResp.CheckoutRequestID } })
    });
  }
  return new Response(JSON.stringify(bodyResp), { status: resp.ok ? 200 : 500, headers: { "Content-Type": "application/json" } });
}

// Webhook handler for mpesa confirmations & stk callbacks
async function handleMpesaWebhook(req: Request, env: Env) {
  const raw = await req.text();
  let json;
  try { json = JSON.parse(raw); } catch(e) { json = raw; }
  await writeRawWebhookToSupabase(env, "mpesa", json, (Object.fromEntries(req.headers.entries())));
  // accept quickly
  // Additional logic: verify fields, idempotency, write to payments table
  // Example: if STK callback:
  if (json.Body && json.Body.stkCallback) {
    const cb = json.Body.stkCallback;
    const checkoutId = cb.CheckoutRequestID;
    const resultCode = cb.ResultCode;
    const resultDesc = cb.ResultDesc;
    // idempotency key per checkoutId
    const idempKey = `mpesa::checkout::${checkoutId}`;
    const exists = await env.KV_IDEMPOTENCY.get(idempKey);
    if (exists) {
      return new Response(JSON.stringify({ ResultCode: 0, ResultDesc: "Accepted (duplicate)" }), { status: 200, headers: { "Content-Type": "application/json" } });
    }
    await env.KV_IDEMPOTENCY.put(idempKey, "1", { expirationTtl: 60 * 60 * 24 * 7 }); // keep 7 days

    if (resultCode === 0) {
      // parse amount and MpesaReceiptNumber from callback callbackMetadata
      let amount = null, mpesaReceipt = null, phone = null;
      const items = (cb.CallbackMetadata?.Item || []);
      for (const it of items) {
        if (it.Name === "Amount") amount = it.Value;
        if (it.Name === "MpesaReceiptNumber") mpesaReceipt = it.Value;
        if (it.Name === "PhoneNumber") phone = it.Value;
      }
      // update payments table by CheckoutRequestID OR reference matched in metadata
      await fetch(`${env.SUPABASE_URL}/rest/v1/payments`, {
        method: "PATCH",
        headers: {
          "apikey": env.SUPABASE_SERVICE_ROLE,
          "Authorization": `Bearer ${env.SUPABASE_SERVICE_ROLE}`,
          "Content-Type": "application/json"
        },
        body: JSON.stringify({
          provider_tx_id: mpesaReceipt,
          status: "VERIFIED",
          received_at: nowIso(),
          metadata: { ...cb }
        })
      });
    } else {
      // failure -> mark payment failed
      await fetch(`${env.SUPABASE_URL}/rest/v1/payments`, {
        method: "PATCH",
        headers: {
          "apikey": env.SUPABASE_SERVICE_ROLE,
          "Authorization": `Bearer ${env.SUPABASE_SERVICE_ROLE}`,
          "Content-Type": "application/json"
        },
        body: JSON.stringify({
          status: "FAILED", metadata: { ...cb }
        })
      });
    }

    return new Response(JSON.stringify({ ResultCode: 0, ResultDesc: "Accepted" }), { status: 200, headers: { "Content-Type": "application/json" } });
  }

  // Fallback quick response
  return new Response(JSON.stringify({ ResultCode: 0, ResultDesc: "Accepted (raw logged)" }), { status: 200, headers: { "Content-Type": "application/json" } });
}

export default {
  async fetch(request: Request, env: Env) {
    const url = new URL(request.url);
    if (request.method === "POST" && url.pathname.endsWith("/initiate")) {
      return handleStkInitiate(request, env);
    }
    if (request.method === "POST" && url.pathname.includes("/webhooks/")) {
      return handleMpesaWebhook(request, env);
    }
    return new Response("ok");
  }
};

"use client";
import React from "react";

type Form = {
  amount: number;
  phone: string;
  name?: string;
  method: "mpesa" | "card" | "mchanga";
};

export default function DonateForm() {
  const { register, handleSubmit, watch } = useForm<Form>({ defaultValues: { method: "mpesa" } });
  async function onSubmit(data: Form) {
    const res = await fetch("/api/payments/initiate", { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify(data) });
    const json = await res.json();
    if (!res.ok) {
      alert("Error: " + (json?.message || res.statusText));
      return;
    }
    if (data.method === "mpesa") {
      // show waiting modal / poll status
      alert("STK initiated. Please complete on your phone.");
    } else if (data.method === "card") {
      // redirect to stripe checkout (returned in json)
      window.location.href = json.checkoutUrl;
    } else {
      // mchanga -> redirect
      window.open(json.mchangaUrl, "_blank");
    }
  }
  return (
    <form onSubmit={handleSubmit(onSubmit)} className="space-y-3 max-w-md">
      <input type="number" {...register("amount", { valueAsNumber: true })} placeholder="Amount KES" className="input" />
      <input type="text" {...register("phone")} placeholder="Phone 2547..." className="input" />
      <input type="text" {...register("name")} placeholder="Full name (optional)" className="input" />
      <select {...register("method")} className="select">
        <option value="mpesa">M-Pesa</option>
        <option value="card">Card (Stripe)</option>
        <option value="mchanga">M-Changa</option>
      </select>
      <button className="btn btn-primary">Donate</button>
    </form>
  );
}

web/app/api/payments/initiate/route.ts — Vercel serverless / Next.js server API (Node)

import { NextResponse } from 'next/server';
import { supabaseAdmin } from '@/lib/supabaseClient';
import fetch from 'node-fetch';

export async function POST(req: Request) {
  const body = await req.json();
  const { amount, phone, method, name } = body;
  if (!amount || !phone) return new Response(JSON.stringify({ message: 'missing' }), { status: 400 });

  if (method === 'mpesa') {
    // Create pending record
    const insert = await supabaseAdmin.from('payments').insert({
      provider: 'mpesa', provider_tx_id: null, reference: `AFLEWO-${Date.now()}`, amount, donor_phone: phone, status: 'PENDING', metadata: {}
    }).select().single();
    if (insert.error) return new Response(JSON.stringify({ message: insert.error.message }), { status: 500 });
    // Call your cloudflare worker or Daraja directly from server - example: call your Worker STK endpoint
    const workerUrl = process.env.MPESA_WORKER_URL + "/initiate";
    const resp = await fetch(workerUrl, { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ amount, phone, reference: insert.data.reference })});
    const json = await resp.json();
    return NextResponse.json({ ok: true, checkout: json });
  }

  if (method === 'card') {
    // Create stripe checkout session (not included here)
    return NextResponse.json({ ok: false, message: "stripe not configured" });
  }

  if (method === 'mchanga') {
    // return mchanga link (manual)
    return NextResponse.json({ ok: true, mchangaUrl: "https://www.mchanga.com/your-campaign" });
  }

  return NextResponse.json({ ok: false });
}

## 4) Airtight backend verification procedures (concrete)

Follow **three layers**: *Authentication*, *Idempotency & atomic DB writes*, *Reconciliation & retention*.

### A. Authentication & trust of callbacks

* **Daraja OAuth**: server obtains access token using consumer key/consumer secret. Store token securely in env and refresh when near expiry. Never expose these to client. 
* **Webhook verification**: if provider signs webhooks, verify signature header (HMAC) using provider shared secret. If no signature (some banks), enforce additional checks: source IP whitelist, TLS client certificate, or require provider to sign. Log raw payload for audit. 
* **TLS only:** Require HTTPS, strong ciphers, certificate monitoring.

### B. Idempotency & race conditions

* Every incoming callback must include a *provider transaction id* (e.g., `MpesaReceiptNumber` or Stripe `id`). Use that as the idempotency key. DB writes must be wrapped in a single transaction with `INSERT ... ON CONFLICT DO NOTHING/UPDATE` keyed on that id. Sample SQL schema below. 

```sql
CREATE TABLE payments (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  provider text NOT NULL,
  provider_tx_id text NOT NULL UNIQUE, -- e.g., MpesaReceiptNumber or Stripe id
  status text NOT NULL,
  amount numeric(12,2) NOT NULL,
  currency text NOT NULL,
  metadata jsonb,
  received_at timestamptz,
  created_at timestamptz DEFAULT now()
);
```

Server logic (pseudocode):

```ts
// Express example for webhooks
app.post('/webhooks/mpesa/confirmation', async (req, res) => {
  const payload = req.body;
  const txId = payload.MpesaReceiptNumber;
  await db.query(
    `INSERT INTO payments(provider, provider_tx_id, status, amount, currency, metadata, received_at)
     VALUES($1,$2,$3,$4,$5,$6,$7)
     ON CONFLICT (provider_tx_id) DO UPDATE
       SET status = EXCLUDED.status
     RETURNING *;`,
    ['mpesa', txId, payload.ResultDesc, payload.Amount, 'KES', payload, now()]
  );
  res.json({ ResultCode: 0, ResultDesc: 'Accepted' }); // quick response to provider
});
```

### C. Quick 200 to provider — async verification & processing

* **Respond fast** to provider (within 2–5 seconds with success code). Do heavy work (email, notification, ledger posting) asynchronously via job queue (BullMQ). This prevents timeouts and retries. 

### D. Full reconciliation

* **Daily reconciliation job**:

  * Pull bank statements (CSV) & M-Pesa Paybill statement (CSV/API).
  * Compare provider transaction ids and amounts with `payments` table.
  * Create exceptions report: missing confirmations, mismatched amounts, reversals.
  * Keep audit trail of reconciliation actions (who resolved, notes). 

### E. Webhook replay / retry safety

* Store raw webhook payloads and headers in `raw_webhooks` table with status `pending/processed/failed`. If processing fails, flag and retry with exponential backoff. Never drop raw payloads: they're evidence.

## 8) Webhook verification sample (Node.js / Express / TypeScript)

```ts
import crypto from 'crypto';

function verifySignature(rawBody: Buffer, headerSignature: string, secret: string) {
  const hmac = crypto.createHmac('sha256', secret);
  hmac.update(rawBody);
  const expected = hmac.digest('hex');
  return crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(headerSignature));
}

app.post('/webhook/stripe', express.raw({type: 'application/json'}), (req, res) => {
  const sig = req.headers['stripe-signature'] as string;
  if(!verifySignature(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET)) {
    res.status(400).send('Invalid signature');
    return;
  }
  // parse and process...
});
```
